# PRD — everyone-play-safety

문서 버전: 0.1
작성일: 2026-04-13
대상 행사: 2026-05-02 (디데이 D-19)
개발 마감: 2026-04-19 (D-13)

---

## 1. 배경과 문제 정의

기존 `everyone-play_v2.0`는 자체 FFmpeg 송출 + 워커 에이전트 + 오버레이 렌더링을 모두 직접 구현했다. 그 결과:

- 하드웨어/플랫폼 다양성(Windows dshow, NDI, Intel/AMD/NVIDIA GPU, 캡처카드 중복 이름, Mac avfoundation)을 모두 직접 처리해야 했고, 각 케이스에서 버그가 발생
- 하드웨어 추상화 레이어를 우리가 다시 만드는 격이 됨
- 행사 직전에도 새로운 하드웨어 조합에서 터지는 시나리오가 우려됨

**해결 방향:** 송출 책임을 OBS Studio에 위임하고, 자체 앱은 우리만 할 수 있는 일(대회 운영 로직)에 집중한다.

## 2. 제품 목표

### 핵심 목표 (Must)

1. **각 코트 라이브 송출** — 각 코트 1대 카메라 영상이 YouTube에 끊김 없이 송출
2. **선수별 클립 추출 및 전달** — 히트 종료 후 자동으로 해당 구간 클립 생성, 참가자에게 SMS로 전달
3. **실시간 오버레이** — 송출 영상에 코트명/히트번호/선수명/타이머 표시
4. **히트 운영 콘솔** — 관리자가 히트 시작/종료, 참가자 배정을 빠르게 처리

### 부차 목표 (Should)

- OBS 연결 상태/녹화 상태/스트리밍 상태 실시간 모니터링
- 카톡 알림톡 발송 (SMS 외 추가 채널)
- 클립 YouTube 자동 업로드

### 비목표 (Out of Scope, 이번 행사)

- 자체 스트리밍 송출
- 6대 이상 코트 동시 운영 (이번 행사 6코트 한정)
- 다중 카메라 스위칭 / PTZ 자동 추적
- 모바일 앱 / 시청자 인터랙션
- 클립 자동 편집 (BGM, 자막, 인트로/아웃트로)

## 3. 사용자

### 운영자 (관리자)
- 1명이 6코트 전체 모니터링
- 코트별 보조 1명씩 배치 가능 (필수 아님)
- 비개발자도 조작 가능해야 함
- 행사 당일 새 기능 학습 불가 — 사전 리허설로만 익숙해질 시간 확보

### 참가자 (선수)
- 자기 경기 직후 SMS로 클립 링크 수신
- 별도 앱/로그인 불필요

## 4. 기능 요구사항

### F1. 이벤트/코트 관리 (기존 재사용)

- 이벤트 생성·수정·조회
- 코트(1~6) 정의 및 OBS 인스턴스 매핑
- 참가자 CSV 일괄 업로드 (BOM-safe, 한글 헤더)

### F2. OBS 연동 (신규)

- 코트별 OBS 인스턴스에 WebSocket 연결 (OBS 28+ 내장)
- 백엔드 → OBS 명령:
  - 녹화 시작/중지
  - 스트리밍 시작/중지
  - 씬 전환 (필요 시)
- OBS → 백엔드 이벤트 수신:
  - 녹화 시작 시각 (타임스탬프 동기화 기준)
  - 드롭 프레임 / 비트레이트 / 연결 상태
- OBS 연결 실패 시 대시보드 즉시 경고

### F3. 오버레이 (신규)

- 정적 HTML 페이지 (`overlay.html`) 제공
- OBS 브라우저 소스가 이 페이지를 로드 (URL: `http://api-host/overlay/?court=N`)
- 페이지가 백엔드에 polling 또는 SSE로 현재 상태 구독
- 표시 요소:
  - 코트 라벨 (예: "코트 1")
  - 진행 상태 (대기/LIVE/완료)
  - 히트 번호
  - 선수 정보 (이름, 팀)
  - 타이머 (히트 시작 후 경과 시간)
  - LIVE 뱃지

### F4. 히트 관리 (기존 보강)

- 히트 시작 버튼 → 백엔드가 OBS 녹화 시작 명령 + 시작 시각 DB 저장 + 오버레이 상태 갱신
- 히트 종료 버튼 → 백엔드가 종료 시각 저장 + 클립 추출 큐에 작업 등록
- 히트 시작/종료 시각은 OBS 녹화 시작 시각 기준 **오프셋**으로 함께 저장 (클립 추출 정확도 보장)

### F5. 클립 추출 (신규)

- 히트 종료 트리거 → 클립 워커가 OBS 녹화 파일에서 해당 구간 추출
- 추출 도구: FFmpeg (단순 `-ss` `-to` `-c copy` 컷)
- 추출 결과 저장 위치: 일단 로컬 파일 시스템 (TBD: 행사 후 전달 방식 결정)
- 추출 완료 후 SMS 발송 트리거

### F6. SMS 발송 (기존 재사용)

- NHN Cloud SMS API 연동 (legacy backend 모듈 재사용)
- 메시지 템플릿: "{이름} 선수님, 경기 영상이 준비되었습니다. {링크}"
- 발송 상태 DB 기록 (재발송 가능)

### F7. 모니터링 콘솔 (신규)

- 대시보드 메인에 6코트 카드:
  - OBS 연결 상태 (✅/⚠️/❌)
  - 녹화 상태 (녹화 중 / 정지)
  - 스트리밍 상태 (송출 중 / 정지 / 드롭)
  - 현재 히트 정보
  - 누적 처리 히트 수

## 5. 비기능 요구사항

### NF1. 안정성

- 행사 8시간 연속 운영 가능
- 단일 코트 장애가 다른 코트에 영향 없음
- 재시작 시 진행 중 이벤트 상태 복원 가능

### NF2. 성능

- 히트 종료 → SMS 발송까지 5분 이내 (클립 추출 포함)
- 대시보드 응답 시간 < 1초
- OBS 명령 응답 < 500ms

### NF3. 운영성

- 한 번 세팅 후 비개발자 운영 가능
- 코트 추가/제거 < 5분
- 장애 시 어떤 코트가 문제인지 즉시 식별 가능

### NF4. 데이터 안정성

- 히트 시작/종료 시각은 즉시 DB 커밋
- OBS 녹화 파일은 로컬 디스크에 보존 (행사 후 백업)

## 6. 제약사항

- 개발 인력: 1명 + 본부장 협업
- 개발 기간: 7일 (4/13 ~ 4/19)
- 리허설 기간: 1주 (4/20 ~ 4/26)
- 하드웨어:
  - 현재 가용: 랩탑 3대 + 카메라 3대
  - 행사 당일: 랩탑 6대 + 카메라 6대 (3대 추가 확보 예정)
- 네트워크: 행사장 무선 + 유선 백업
- 시스템: Windows 10/11 (랩탑은 모두 Windows)

## 7. 성공 지표

### 개발 단계 (4/19까지)
- 3코트 시뮬레이션에서 30분 연속 송출 + 히트 5개 처리 + SMS 발송 성공

### 리허설 단계 (4/26까지)
- 6코트 실제 하드웨어로 1시간 연속 송출 + 히트 30개 처리 무중단

### 행사 당일 (5/2)
- 6코트 8시간 운영 무중단
- 송출 끊김 < 1% (코트당 5분 이내)
- 클립 누락 0건
- SMS 발송 성공률 > 95%
